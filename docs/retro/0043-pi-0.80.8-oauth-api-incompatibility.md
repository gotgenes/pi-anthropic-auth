# Retro: #43 — Pi 0.80.8 OAuth API incompatibility

## Stage: Planning (2026-07-17T00:00:00Z)

### Session summary

Transformed the pre-existing analysis doc `docs/plans/0043-*.md` into the formal implementation plan for restoring `/login` on Pi 0.80.8+.
Confirmed the fix direction (Option A — drop the `oauth` override, delegate login/refresh to Pi's built-in `anthropicOAuth`) with the operator via two `ask_user` rounds, the second after explaining the consequences of losing `mergeRefreshedCredentials`.
The plan is a removal + dependency-floor raise + docs pass, so it dispatches to `/build-plan`, not `/tdd-plan`.

### Observations

- **Option A vs B was a genuine `ask_user` gate.** Even though the operator authored the issue recommending Option A, the trade-off (dropping `mergeRefreshedCredentials`) is breaking and fallback-semantics-shaped, so I surfaced it. The operator chose Option A, accepting the refresh-rotation risk; Option B (vendoring the flow) and an upstream follow-up were both explicitly declined and recorded as Non-Goals.
- **Consequence of losing `mergeRefreshedCredentials`, verified from source:** built-in `refreshAnthropicToken` (0.80.8) sets `refresh: data.refresh_token` verbatim; an omitted field becomes `undefined` and breaks the next refresh until manual re-login. Only bites if Anthropic ever omits a rotated token (RFC 6749 §6 permits it; observed behavior always rotates).
- **Breaking-change classification:** the peer-floor raise to `>=0.80.8` follows Issue #40's precedent (floor raise released as a major). The plan puts `fix!:` + `BREAKING CHANGE:` on the floor-raise commit (step 2), not the removal commit (step 1).
- **Step ordering matters for typecheck:** the module deletion must land before/with the devDep bump. Bumping to `0.80.8` while `src/anthropic-oauth.ts` still imports the removed functions from the now-types-only `/oauth` subpath would fail `tsc`. Step 1 deletes on `0.80.2` (functions still present); step 2 bumps and re-validates against `0.80.10`.
- **CHANGELOG is release-please-generated** — no manual `[Unreleased]` edit; the Conventional Commit subjects/footers drive it. Corrected the earlier issue-body assumption that a manual CHANGELOG entry was needed.
- **Operator-confirmed doc path fix folded in:** `~/development/pi-mono` references in `AGENTS.md` are wrong; the real clone is `~/development/pi/pi`. Added to the docs step (step 3) since `AGENTS.md` is already being edited.
- **Latest pi-ai is `0.80.10`** — chosen as the devDep pin target.
- Historical plan/retro records that mention the override (`0031`, `0033`, `0037`, `gap-analysis-and-next-steps.md`, retro `0027`) are left as-is (Non-Goals) — point-in-time records.

## Stage: Final Retrospective (2026-07-17T16:26:28Z)

### Session summary

Shipped Issue #43 as `v2.0.0` (release-please PR #41, merged rebase), but only after the ship session surfaced a plan gap the earlier stages missed.
The operator reported `/login` still failing after upgrading Pi; the diagnosis that followed uncovered that Pi's `registerProvider` merges configs (preserving `undefined` keys), so omitting `oauth` does not clear a broken `oauth` registered by a stale copy that loads alongside the fixed one.
The fix was validated live by isolating to the fixed copy and completing a real `/login` browser leg, then the migration hazard was documented (`8f5618b`) before release.

### Observations

#### What went well

- **Source-driven root-cause once prompted.** Reading `ModelRuntime.registerProvider` in the pinned `~/development/pi/pi` clone revealed the merge-preserves-`undefined` semantics that explained why a loaded fixed copy still crashed. This turned a hand-wavy "stale install" story into a precise, documented mechanism.
- **Clean live isolation test.** Removing the global `packages[]` entry (leaving only the local fixed copy via the repo `.pi/settings.json` `"../"` entry) both unblocked the operator and served as the definitive `/login` validation the automated suite cannot provide.
- **`ask_user` used well at the real decision boundary.** The simple-omit vs defensive-`unregister` choice was surfaced with concrete trade-offs once the merge hazard was understood.

#### What caused friction (agent side)

- `premature-convergence` / `missing-context` — On the operator's first "still failing" report I converged immediately on "the installed npm copy is stale and unpublished," built an elaborate table around it, and checked only the global `~/.pi/agent/settings.json` — never reading the repo `.pi/settings.json` that the operator had to point me to.
  Impact: the operator had to push back ("Local package should work because it overrides... are we confident our fix works? What troubleshooting have you done?") to redirect me to the actual root cause; user-caught, cost a round of confident-but-incomplete explanation before the real investigation started.
- `missing-context` (planning-stage origin) — The plan's Design Overview asserted "with `oauth` omitted, it uses the built-in `anthropicOAuth`" from reading `composeOAuthAuth` alone, but never checked `registerProvider`'s merge semantics. The mechanism that actually governs whether `oauth` is present was left unverified.
  Impact: the fix had an undocumented migration hazard; surfaced only under ship-session pushback and required an extra `docs:` commit (`8f5618b`).
- `instruction-violation` (self-identified, but slowly) — Roughly eight consecutive `ask_user` tool calls failed because option objects omitted the required `value` (and sometimes `label`) fields, even though the system prompt states "always include a non-empty machine-readable `value` for every option." I retried nearly one field at a time instead of reading the full validation error and fixing every named field at once.
  Impact: ~8 wasted tool calls and conversation clutter across two questions; no rework to shipped artifacts.
- `wrong-abstraction` (overconfidence) — At the end of the build stage I reported "done, all green, ready to ship" on green `check`/`test`/`lint`/`fallow`, despite the build retro itself noting the live `/login` leg was unrun. The one runtime path that mattered was deferred to ship.
  Impact: confidence outran evidence; the operator's "what troubleshooting have you done?" was the correction.

#### What caused friction (user side)

- The repo's two-copy load model (local `"../"` source + a `packages[]` npm entry, plus a separate global `packages[]` entry) is the crux of the whole failure, and it lived only in `.pi/settings.json` files. Surfacing "note that a global install of this package also loads and can clobber the local copy" at the first report would have short-circuited my stale-install detour. Framed as opportunity: this is now captured as a durable gotcha so it need not be re-explained.

### Diagnostic details

- **Escalation-delay:** the eight-call `ask_user` failure streak is the clearest "should have stopped and re-read after the 2nd identical validation error" signal. The stale-install detour was shorter (~2-3 tool batches) and was ended by the operator, not self-caught.
- **Unused-tool / missing step:** no subagent was needed — the missing move was a plain `read` of the repo `.pi/settings.json` (and recognizing the global `packages[]` entry) as step one of "which copy loads." I reached for greps of installed `node_modules` before reading the settings that decide loading.
- **Feedback-loop gap:** the automated suite ran incrementally during build (good), but the *runtime* login validation ran only at ship time. For provider-registration/OAuth changes, a green `pi -p` prompt does not exercise `/login`; the interactive leg is a distinct verification that belongs in the build "done" bar, not after it.
- **Model correlation:** primary work ran on `claude-opus-4-8` (appropriate for the judgment-heavy diagnosis); the `pre-completion-reviewer` dispatch in the build stage was a reasonable use of a fresh-context reviewer. No model/task mismatch found.

## Stage: Post-Release Incident — stale install on refresh (2026-07-17T16:26:28Z)

### Session summary

Shortly after `v2.0.0` shipped, the operator hit `Error: OAuth refresh failed for anthropic` and suspected the dropped `mergeRefreshedCredentials` guard ("threw the baby out with the bathwater").
Diagnosis proved otherwise: the global install was still `1.0.0` (broken `oauth` override), its `refreshToken` calls the removed `refreshAnthropicToken`, and the `registerProvider` merge preserved that `oauth` over the local `2.0.0` copy — the same merge-clobber hazard, now on the refresh path (`resolve.ts:102` wraps `oauth.refresh`).
The stored `.anthropic.refresh` token was intact throughout, so the accepted refresh-rotation risk had not materialized; a `pi install npm:@gotgenes/pi-anthropic-auth@latest` upgrade fixed it and a live request returned `REFRESH_OK`.

### Observations

- **The breaking (major) release does not auto-propagate.** The global install was pinned `^1.0.0` (npm's default caret from the original `pi install`), and `v2.0.0` is outside that range, so `pi update`/`pi update --extensions` treat `1.0.0` as "up to date" and never cross the major. `pi install npm:<pkg>@2.0.0` reported "up to date" as well until an explicit version/tag forced the rewrite; `@latest` is the canonical cross-the-major command (it rewrites the pin to `^2.0.0`). Mental model surfaced for the operator: `pi update` = stay within your major; `pi install …@latest` = adopt a new major.
- **`missing-context` (again, but faster to correct).** First instinct was still to name the stale-install story, but this time I gated it on evidence — read `/anthropic-auth:status` (local `2.0.0`), the global `package.json` (`1.0.0` + broken deep import), `resolve.ts:102`, and the intact `.anthropic.refresh` token — before asserting. The token-shape check is what definitively ruled out the `mergeRefreshedCredentials` regression. Impact: no user rework; correct call on the first framed answer.
- **Migration-communication gap in the ship close comment.** The issue #43 close comment said "update or remove any stale copy" but did not give the exact command or warn that `pi update` silently holds back the major. Anyone hitting `OAuth refresh failed` after upgrading would be stuck on the same subtlety.

### Changes made

1. `AGENTS.md` — extended the `registerProvider` Merges gotcha with the major-bump upgrade note: `pi update` stays within the `^` range, so cross a major with `pi install npm:<pkg>@latest` (Refs #43).
2. Issue #43 — posted a follow-up comment documenting the `OAuth refresh failed` / stale-copy symptom and the `pi install npm:@gotgenes/pi-anthropic-auth@latest` fix.
3. `docs/retro/0043-pi-0.80.8-oauth-api-incompatibility.md` — added this Post-Release Incident stage entry.

Considered but not implemented: a Pi upstream write-up proposing `pi update` surface cross-major extension upgrades (operator did not select it this round).
