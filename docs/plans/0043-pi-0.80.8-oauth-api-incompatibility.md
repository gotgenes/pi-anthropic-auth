# Re-enable pi-anthropic-auth on Pi 0.80.8+ by Dropping the OAuth Override

## Release Recommendation

**Release:** ship independently

This fix is not part of any architecture-roadmap release batch.
It restores a completely broken `/login` flow on the current Pi generation, so it should ship on its own as soon as it is green.

## Problem Statement

On Pi 0.80.8 and later, running `/login` for the Anthropic subscription crashes immediately:

```text
Error: Failed to login to Anthropic: (0 , _oauth.loginAnthropic) is not a function
```

Pi 0.80.8 removed the public low-level OAuth functions from `@earendil-works/pi-ai/oauth`; that subpath now re-exports types only.
Our `src/anthropic-oauth.ts` deep-imports `loginAnthropic` and `refreshAnthropicToken` from it, so both bindings resolve to `undefined`.
The extension still loads (the module exists; only the bindings are undefined), but the first `/login` call invokes `undefined(...)` and throws.
Our `peerDependencies` floor of `>=0.80.0` admits 0.80.8+, which is the trap: 0.80.0–0.80.7 still work.

The chosen fix is **Option A** from the prior analysis: drop the `oauth` override entirely and let Pi's built-in `anthropicOAuth` handle login and refresh, keeping only the `streamSimple` transport wrapper.
The operator confirmed this direction, accepting the loss of `mergeRefreshedCredentials` on the judgment that Anthropic reliably rotates a fresh `refresh_token` on refresh.

## Goals

- Restore `/login anthropic` on Pi 0.80.8+ by removing the deep import of the now-private `loginAnthropic`/`refreshAnthropicToken`.
- Delegate login and refresh to Pi's built-in `anthropicOAuth` (via `composeModelProvider`'s fallback when no `oauth` override is supplied).
- Preserve the `streamSimple` transport wrapper and all its request shaping unchanged — it remains the extension's reason to exist.
- Raise the peer floor to `@earendil-works/pi-ai >=0.80.8` and `@earendil-works/pi-coding-agent >=0.80.8`, and bump the dev dependency pins to `0.80.10` so CI validates against the new API.
- Update `AGENTS.md`, `docs/architecture.md`, and `.pi/skills/anthropic/SKILL.md` to remove references to the deleted OAuth override and refresh fallback.

This change is **breaking**: it raises the host peer floor (users on 0.80.0–0.80.7 lose declared support) and drops the `mergeRefreshedCredentials` refresh-token guard.
The floor-raise commit uses `fix!:` with a `BREAKING CHANGE:` footer, matching the precedent set by Issue #40's floor raise (released as the 1.0.0 major).

## Non-Goals

- Vendoring the OAuth flow to retain `mergeRefreshedCredentials` (Option B) — explicitly rejected for this issue after weighing the maintenance cost against the low probability of Anthropic omitting a rotated `refresh_token`.
- Filing an upstream request for Pi's built-in refresh to preserve an omitted `refresh_token` — considered and declined for this issue.
- Any change to `src/host-transport.ts`, `src/oauth-transport.ts`, `src/request-shaping.ts`, `src/system-prompt-shaping.ts`, `src/constants.ts`, or `src/diagnostics.ts`.
  The transport wrapper is still required: Pi's built-in Anthropic transport emits Claude-Code identity headers and tool-name normalization for OAuth tokens, but not the body-level `x-anthropic-billing-header` system block, the assistant `tool_use`-trailing-content split, or the system-prompt sanitization.
- Rewriting historical plan/retro records (`docs/plans/0031-*`, `0033-*`, `0037-*`, `docs/plans/gap-analysis-and-next-steps.md`, `docs/retro/0027-*`) that mention the override — they are point-in-time records and stay as written.

## Background

Relevant modules and how they relate:

- `src/anthropic-oauth.ts` — exports `anthropicOAuthOverride` (an `ExtensionOAuthConfig`-shaped object with `login`/`refreshToken`/`getApiKey`/`usesCallbackServer`) and the `mergeRefreshedCredentials` helper.
  `login` and `getApiKey` are pure passthroughs of built-in behavior; `refreshToken` calls `refreshAnthropicToken` then `mergeRefreshedCredentials`.
  The deep import `import { loginAnthropic, refreshAnthropicToken } from "@earendil-works/pi-ai/oauth"` is the single point of breakage.
- `src/index.ts` — imports `anthropicOAuthOverride` and passes it as `oauth` to `pi.registerProvider("anthropic", { oauth, api, streamSimple })`.
  This is the only consumer of the override in `src/`.
- `test/anthropic-oauth.test.ts` — imports `loginAnthropic` from `@earendil-works/pi-ai/oauth` and `anthropicOAuthOverride`/`mergeRefreshedCredentials` from `#src/anthropic-oauth`.
  Every test in this file exercises code that is either being deleted (the override, `mergeRefreshedCredentials`) or is now Pi-owned (`loginAnthropic`).

What is confirmed not broken (scoping evidence, from the pi-mono clone at `~/development/pi/pi`):

1. **The override's object shape is still accepted.** `packages/coding-agent/src/core/provider-composer.ts` ships `adaptOAuth(config)`, and `ProviderConfig.oauth` in `packages/coding-agent/src/core/extensions/types.ts` still takes the old `OAuthLoginCallbacks` shape.
   So the composition path itself did not break — only the deep import did.
2. **`composeOAuthAuth` falls back to the built-in when `oauth` is omitted.**
   It does `const oauth = extension?.oauth ? adaptOAuth(extension.oauth) : base?.auth.oauth;`, so omitting `oauth` uses `anthropicOAuth` (lazy-loaded from `packages/ai/src/auth/oauth/anthropic.ts`).
3. **The `streamSimple` overlay still wins for `anthropic-messages` models**, and `composeApiKeyAuth` still composes the ambient `ANTHROPIC_API_KEY` path — so API-key and OAuth both keep working.
4. **The built-in `loginAnthropic` already supports the manual redirect-URL / bare-code paste** (`parseAuthorizationInput`), so dropping our `usesCallbackServer: true` override (Issue #27) does not regress manual-code login.

The one behavior we lose is `mergeRefreshedCredentials`.
The built-in `refreshAnthropicToken` (0.80.8, `packages/ai/src/auth/oauth/anthropic.ts`) sets `refresh: data.refresh_token` verbatim; if a refresh response omits `refresh_token`, the stored value becomes `undefined` and the next refresh fails until a manual re-login.
Per RFC 6749 §6 a compliant server may omit it, but Anthropic's endpoint has rotated a fresh token on every observed refresh — the accepted risk.

AGENTS.md constraints that apply:

- Keep the override thin and preserve built-in behavior by default — this change moves *toward* that principle by deleting local code in favor of upstream.
- Use `pnpm`; when a `package.json` dependency changes, run `pnpm install` and commit the updated `pnpm-lock.yaml` in the same commit (CI uses `--frozen-lockfile`).
- The CHANGELOG is generated by release-please from Conventional Commits; do not hand-edit `CHANGELOG.md`.
  The `fix!:` subject and `BREAKING CHANGE:` footer drive the release notes and the major bump.

## Design Overview

The registration collapses from two overrides to one:

```typescript
// src/index.ts — after
pi.registerProvider("anthropic", {
  api: "anthropic-messages",
  streamSimple: createAnthropicOAuthStreamSimple(streamSimpleAnthropic),
  // no `oauth`: Pi's built-in anthropicOAuth handles login + refresh.
});
```

`composeModelProvider` resolves the OAuth auth as `extension?.oauth ? adaptOAuth(extension.oauth) : base?.auth.oauth`.
With `oauth` omitted, it uses the built-in `anthropicOAuth`, whose `login`/`refresh`/`toAuth` cover exactly what our passthrough override did — minus the `mergeRefreshedCredentials` guard we are intentionally dropping.

No new collaborator is introduced; this is a deletion plus a call-site simplification, so there is no Tell-Don't-Ask or Law-of-Demeter surface to sketch.

Edge cases:

- **API-key requests** are unaffected — `composeApiKeyAuth` still wires `ANTHROPIC_API_KEY` from `base.auth.apiKey`, and the transport wrapper's `sk-ant-oat` gate short-circuits to pass-through for non-OAuth tokens.
- **Manual-code login** (paste redirect URL or bare code) is preserved by the built-in flow.
- **Refresh-token omission** is the accepted-risk edge case: if it ever occurs, the user re-runs `/login anthropic`.

## Test Impact Analysis

This is a removal, not an extraction, so the analysis is about what leaves rather than what new coverage is enabled:

1. **No new unit tests are enabled.** The deleted behavior (login passthrough, `mergeRefreshedCredentials`) moves to Pi-owned code; the OAuth browser leg is not unit-testable on our side.
2. **`test/anthropic-oauth.test.ts` becomes entirely redundant** and is deleted with the module.
   Its `loginAnthropic` tests exercised code that is now private to Pi; its `mergeRefreshedCredentials`/`usesCallbackServer` tests exercised code that no longer exists.
3. **The surviving suites must stay green and unchanged**: `test/oauth-transport.test.ts`, `test/request-shaping.test.ts`, `test/system-prompt-shaping.test.ts`, and `test/pi-anthropic-ordering-experiment.test.ts`.
   They genuinely exercise the transport wrapper, which this change does not touch.
4. **Manual verification**: a live `/login anthropic` smoke test against a real Anthropic subscription on Pi 0.80.8+ is the only coverage for the delegated login path.
   Follow the `pi-cli-repro` workflow; the browser leg cannot be automated.

## Invariants at Risk

The transport-wrapper shaping invariants must not regress, even though this change does not edit those files:

- Billing-header injection, `system[]` block ordering, and cache-control handling — pinned by `test/request-shaping.test.ts`.
- Anchor-based system-prompt sanitization (identity replacement, tool-snippet/guideline/appended-content preservation) — pinned by `test/system-prompt-shaping.test.ts`.
- `sk-ant-oat` token gating and `onPayload` composition/delegation — pinned by `test/oauth-transport.test.ts`.
- Assistant `[tool_use…, text]` ordering normalization — pinned by `test/pi-anthropic-ordering-experiment.test.ts`.

Running the full suite after each step confirms none of these regress.

## Risks and Mitigations

- **Risk: Anthropic omits `refresh_token` on a refresh, silently breaking auto-refresh.**
  Mitigation: accepted risk per the operator's decision; recovery is a manual `/login anthropic`.
  The failure mode is documented in this plan and the CHANGELOG breaking-change note so a future re-login report is easy to trace back.
- **Risk: `composeModelProvider` does not actually fall back to the built-in `anthropicOAuth` when `oauth` is omitted, leaving no login handler.**
  Mitigation: verified against `provider-composer.ts` in the pinned source (`extension?.oauth ? adaptOAuth(...) : base?.auth.oauth`); confirmed empirically by the live `/login` smoke test in step 2/verification.
- **Risk: the peer-floor raise surprises users on 0.80.0–0.80.7.**
  Mitigation: `fix!:` + `BREAKING CHANGE:` footer surfaces it in the release notes; the issue documents the `@earendil-works/pi-coding-agent@0.80.7` pin as the only stopgap for those who cannot upgrade.
- **Risk: a stray importer of the deleted module breaks the build.**
  Mitigation: grep confirms `src/index.ts` and `test/anthropic-oauth.test.ts` are the only importers; both are handled in step 1, and `pnpm run check` in steps 1–2 catches any miss.
